# 评论模板库 + 目标帖子筛选指南

---

## 一、评论的本质

**评论不是给原帖作者看的，是给所有看到这条帖子的人看的。**

每条评论的真正目标：让陌生人读完以后，想点进你的主页。

X 的评论排序逻辑：
- 高粉丝账号的评论排名靠前（这是 Tier 1 的劣势）
- 但"获得互动的评论"（有 likes 和 replies 的评论）会被算法推高
- 结论：**Tier 1 账号只有靠评论质量超越账号权重**

---

## 二、如何找到值得评论的帖子

### Tier 1 (0-500粉) — 找"流量洼地"

目标：找到有流量但评论还不多的帖子，在"黄金窗口"占据靠前位置。

**黄金窗口定义**：帖子发布后 30-90 分钟，views 已经上来，但评论 <20 条。

**选帖决策：**

```
评论数 < 10 + 发布 < 30min → ⭐⭐⭐⭐⭐ 最理想（抢头部）
评论数 10-20 + 发布 30-90min → ⭐⭐⭐⭐ 仍然优质
评论数 20-40 + 发布 1-3h → ⭐⭐⭐ 可以
评论数 40-80 + 发布 3-6h → ⭐⭐ 次优
评论数 > 80 或发布 > 6h → ❌ 跳过
```

**作者粉丝数考量：**

| 作者粉丝 | 适合 Tier 1？ | 原因 |
|---------|-------------|------|
| 1K-10K | ⚠️ 太小 | 帖子流量太低，曝光有限 |
| 10K-50K | ✅ 最佳 | 有流量，评论区不会过载 |
| 50K-200K | ✅ 好 | 流量大，但你的评论需要更强才能被看到 |
| 200K+ | ⚠️ 谨慎 | 帖子肯定有流量，但你的评论很容易被淹没 |

**主动找目标帖的三种方法：**

1. **关键词搜索（主动猎帖）**
   - 搜索内容支柱的核心词，点"Latest"排序
   - 找 <2h 的帖子，过滤作者粉丝 10K-200K
   - 这能找到 For You 没推给你的高质量目标

2. **Following tab 浏览**
   - 你关注的中型 KOL 发的新帖子
   - 优点：更精准匹配你的 niche

3. **竞品账号互动区**
   - 看与你定位相似的账号在给谁评论
   - 那些帖子通常也适合你

---

### Tier 2 (500-2000粉) — 找"niche 定位帖"

目标不再是最大化曝光，而是在核心 niche 里持续出现。

**变化：**
- 话题筛选更严格：只评论直接命中核心支柱的帖子（不为了流量而评无关话题）
- 可以接受更高竞争：评论数 <80，发布 <6h
- 可以挑战更大账号（50K-500K），因为账号本身已有一定权重

**主题聚焦策略**

每周选定 1-2 个核心话题，在这些话题下密集评论，而不是分散在各个话题。
这样 X 算法会把你的账号与这些话题关联起来，推给关注这些话题的人。

## 三、评论类型模板

### Type A: 增量价值（最高价值）

**使用条件**：你有真实的相关经历或数据。绝对不要编造。

**结构**：
```
[原帖论点] + [你的具体延伸/反例].
[一句话说明意义].
```

**好的例子（有具体细节，感觉真实）：**
```
We saw this with ~200 early users. The ones who stick aren't the
ones who get it immediately — it's the ones who fail once,
then come back. Something about that first failure locks in the habit.
```

```
Built this exact thing 18 months ago. The hard part wasn't the AI —
it was getting people to check it at all. Solved it with a
digest email, not a dashboard.
```

**差的例子（数字编造，感觉假）：**
```
We tested this at our company — X outperformed Y by 40%.
The trick was focusing on user retention.
```
（任何人都能写这个，没有真实感）

**提示**：如果账号没有相关真实经历 → 用 Type B 或 Type D，不要强行用 Type A。

---

### Type B: 视角补充

**使用条件**：原帖说的是 A，你有一个同样成立但原帖没提到的 B。

**结构**：
```
[原帖观点] works for [适用场景].
But [你的补充] for [另一种场景].
The difference is [关键区分因素].
```

**例子：**
```
This works well for B2C. For B2B the dynamic flips —
buyers want certainty, not delight.
The evaluation cycle is 4-6x longer and price sensitivity is lower.
```

```
True for early-stage. But post-PMF the bottleneck shifts.
It's not "can we build it" — it's "can we explain it in 10 seconds."
Distribution kills more products than execution does.
```

---

### Type C: 善意反驳

**使用条件**：你有具体理由认为原帖结论在某些条件下不成立。
**重要**：必须有具体依据，不能只是"我感觉不一样"。

**结构**：
```
Counterpoint: [具体的不同结论].
[你的依据 — 具体场景或数据].
Might be a [条件/情境] thing.
```

**例子：**
```
Counterpoint: we saw the opposite with our power users.
Heavy users churn when you add too many features —
they came for focus, not a Swiss army knife.
Might be a segment thing.
```

**避免**：
- 反驳只为了显得深刻
- 没有具体依据的"其实不是这样的"

---

### Type D: 问题延伸

**使用条件**：原帖触发了你脑子里一个真实的、更深的问题。

**结构**：
```
This makes me wonder: [由原帖引发的真实问题].
[你的相关观察，表明你不是随便问的].
```

**例子：**
```
This makes me wonder: if AI can handle the "what to do" part,
what's left for the human?
We've been assuming judgment — but I'm less sure what that means now.
```

```
Makes me wonder if the benchmark is the wrong unit entirely.
We track model performance, but never "did the user actually get what they needed."
Those two things diverge pretty often in practice.
```

**避免**：
- 空洞的问题（"But what does this mean for the future?"）
- 显然已经有答案的问题

---

### Type E: Build in Public（真实挣扎）

**使用条件**：原帖描述的情况你真的正在经历，且你有具体的细节可以分享。

**结构**：
```
Going through this right now.
[具体情况 — 必须真实可信]. Still figuring out [具体挑战].
```

**好的例子：**
```
Going through this right now.
We hit our first 100 paying users last week and immediately
the support load tripled. Still figuring out where the triage line is.
```

**差的例子（太模糊，任何人都能写）：**
```
Going through this exact thing right now.
It's tough but we're figuring it out. Still learning.
```

**规则**：如果没有真实相关经历，不要用 Type E。

## 四、评论长度规范

| 类型 | 长度建议 | 原因 |
|------|----------|------|
| Hot take / 反驳 | 1-3 句（60-120字符） | 越简洁越有力 |
| 数据/经历补充 | 2-4 句（100-180字符） | 需要足够上下文 |
| 问题延伸 | 2-3 句 | 问题 + 自己的观察 |
| Build in Public | 2-4 句 | 需要具体细节 |

**超过 5 句就太长了**——读者不会读完，算法也不会优先显示。

---

## 五、评论禁用词 & 反模式

**绝对禁用：**
- "Great post!" / "Love this!" / "So true!"
- "This!" / "Exactly this." / 纯 emoji 回复
- "As a [身份], I think..."
- "DM me if you want to discuss"

**反模式（写出来会减分）：**

| 反模式 | 问题 | 改法 |
|--------|------|------|
| 重复原帖观点 | 没有增量 | 加一个原帖没有的具体细节 |
| 只有问题没有观点 | 显得浅薄 | 问题 + 你的相关观察 |
| 编造数据 | 听起来假 | 如果没有真实数据，用描述性语言 |
| 奉承 + 观点 | "Great point, and also..." | 直接说观点，去掉奉承 |
| 自我推销 | "We're building this at [company]" | 只在真正相关时提，不是广告 |

---

## 六、快速检查流程

每条评论发出前，问三个问题：

1. **独特性**：这条评论如果换成另一个人发，能说吗？如果任何人都能这么写 → 重写
2. **真实性**：有没有编造的细节？如果有 → 换成真实细节，或换类型
3. **吸引力**：陌生人读完以后，会想点我的主页吗？如果不会 → 重写

## 七、Tier 差异总结

| 维度 | Tier 1 (0-500) | Tier 2 (500-2000) |
|------|----------------|-------------------|
| 选帖标准 | views 5K-50K，评论 <30，<2h | views 10K-200K，评论 <80，<6h |
| 话题范围 | 内容支柱内，可以稍宽 | 严格限定核心支柱 |
| 每天目标 | 5-8 条高质量评论 | 3-5 条精准评论 |
| 首要目标 | 被陌生人看到，访问主页 | 强化 niche 定位 |
| 评论类型偏好 | Type A、E（真实感强）| Type A、B、C（权威感强）|
